# 事業別 支出先データ品質スコア — AI評価による再設計

**日時**: 2026-06-16 17:49 (JST)
**対象**: `/quality` ページの事業別支出先データ品質スコア
**目的**: 現行のルール/辞書ベース5軸を、より納得感のある「AI評価2軸＋機械計算2軸」に再設計し、スコアをパイプラインでバッチ事前計算する。

---

## 1. なぜ現行スコアは「納得感がない」のか

現行は完全にルール/辞書ベースの加重5軸（`scripts/score-project-quality.py`）。
ユーザーが特に問題視したのは **軸1（支出先名品質）** と **軸2（CN記入率）**。

| 軸 | 重み | 仕組み | 納得感を損なう構造的問題 |
|----|------|--------|------------------------|
| 軸1 支出先名品質 | 40% | 厳密辞書に名称が**載っているか**の二値 | 「品質」ではなく**辞書カバレッジ**を測っている。辞書未登録の実在企業は `unknown` として分母から除外され、悪質な空白事業がかえって高得点になりうる。名称が正しくても辞書に無ければ低評価。 |
| 軸2 法人番号記入率 | 20% | 法人番号(CN)が埋まっているか | 個人・外国法人・国の機関・海外団体は**そもそもCNを持たない**。CN空欄＝低品質と一律に減点するのは不公平。CNが埋まっている≠支出先が説明可能、でもない。 |
| 軸3 予算・支出バランス | 20% | \|執行−実支出\|/執行 | おおむね妥当だが、年度跨ぎ・タイミング差で正当な乖離も減点。 |
| 軸4 ブロック構造 | 10% | 再委託深度で減点＋金額不整合で減点 | **再委託が深い＝データ品質が悪い、ではない**。深い再委託は構造的複雑さであって記載品質の欠陥ではない。金額不整合の検知は妥当。 |
| 軸5 透明性 | 10% | 不透明キーワード辞書マッチ率 | 妥当だが13ルールのキーワード辞書依存で脆い。軸1と評価対象が重複。 |

**核心**: 軸1・軸2・軸5はいずれも本質的に「**この支出先が誰なのか、第三者が特定・検証できるか**」を別々の脆いルールで近似している。これを意味理解できるAIに統合すれば、辞書カバレッジや一律CN減点の不公平を排し、はるかに納得感のある指標になる。

---

## 2. 設計方針

「国民がこの事業の金の流れを追跡・検証できるか（説明責任・透明性）」を測る、という観点でゼロベース再設計する。

- **AIが判定する軸**: ルールが脆く意味理解が要る部分 → 支出先の特定可能性、使途の説明性
- **機械が計算する軸**: 数字で客観的に決まる部分 → 収支整合性、構造整合性

利用可能な per-recipient データ（`public/data/project-quality-recipients-{year}.json`）は AI 判定に十分な文脈を持つ:
`n`=支出先名 / `c`=法人番号有無 / `o`=不透明フラグ / `a2`=金額 / `role`=事業上の役割 / `cc`=契約概要 / `chain`=委託チェーン。
→ **生CSVを再処理せず、既存JSONだけでAI再評価が可能。**

## 3. 新スコアリング（4軸）

| 新軸 | 種別 | 重み | 測るもの | 入力 |
|------|------|------|----------|------|
| **A. 支出先の特定可能性** | AI（辞書で補助） | 35% | この支出先が具体的に誰で、第三者が実在を確認できるか | 名称・法人番号有無・契約概要・役割 |
| **B. 使途の説明性** | AI | 25% | 何にいくら使ったかが契約概要・役割から理解できるか | 役割・契約概要・金額 |
| **C. 収支の整合性** | 機械 | 25% | 執行額と実支出が説明可能な範囲で一致するか | 執行額・実質支出（ルートブロック合算） |
| **D. 構造の整合性** | 機械 | 15% | ブロック金額の内部整合性・孤立ブロックの有無 | ブロック合計額・支出先合計・孤立ブロック |

AI = A(35)+B(25) = **60%** ／ 機械 = C(25)+D(15) = **40%**。

### 旧→新の対応とアップグレード点

- 旧軸1（辞書）＋旧軸2（CN）＋旧軸5（不透明）→ **A（特定可能性）に統合**。AIが名称＋CN有無＋契約概要を総合し「誰に払ったか検証可能か」を判定。
  - 辞書valid＋CN → 高（特定容易）
  - 不透明名（その他/未定/非公開/個人/プレースホルダ）→ 低
  - 曖昧な中間（辞書未登録の実在企業、CN無しでも契約概要で実体が明確な国の出先機関や海外団体）→ **AIが公平に判定**。「辞書に無い＝除外」「CN無し＝一律減点」の不公平を解消。
- **B（使途の説明性）は新設**。名称が完璧でも契約概要が空白/定型文なら低品質、という説明責任の本丸をAIが判定。
- C: 収支乖離に**許容バンド**を導入し、正当なタイミング差での過剰減点を緩和。
- D: **再委託深度による減点を廃止**（深い再委託は品質欠陥ではない）。金額不整合・孤立ブロックのみ減点。深度はUIに情報として表示。

### AI判定の出力（per 支出先）

各支出先について 0〜3 のレベルで判定:

- **identifiability (0-3)**: 0=特定不能(その他/未定等) / 1=曖昧(辞書未登録かつ文脈薄い) / 2=ほぼ特定可(出先機関・改組大学・文脈で実体明確) / 3=完全特定可(正式名称＋CN or 公的機関)
- **purpose (0-3)**: 0=記載なし / 1=定型・抽象的 / 2=概ね具体的 / 3=具体的で検証可能
- `reason`: 短い根拠（UIの説明・監査用）

レベル→点数: `0→0, 1→40, 2→70, 3→100`。
プロジェクト集計は **金額(a2)加重平均**（金額不明行は件数加重にフォールバック）。

## 4. パイプライン構成

```
score-project-quality.py        … 生CSV→per-recipient行＋機械signal（収支C・構造D・per-recipient文脈）を出力（既存）
        │  project-quality-scores-{year}.json（機械signal）
        │  project-quality-recipients-{year}.json（per-recipient文脈）
        ▼
score-project-quality-ai.py     … 【新規・本設計の中核 / 最終スコアラ】
        │  1. recipients JSON読込
        │  2. ゲーティング＋重複排除（(name,cc,c)タプル単位）
        │  3. OpenRouter(OpenAI互換, openai SDK)でLLM判定 identifiability/purpose（並列・キャッシュ）
        │  4. プロジェクト単位に金額加重集計 → 軸A・軸B
        │  5. 軸C・軸DはJSONのsignalから機械計算
        │  6. 新重み(35/25/25/15)で総合再計算
        │  7. scores JSONを新スキーマで上書き
        ▼
project-quality-scores-{year}.json（最終・UI表示用）
```

### コスト制御

- **重複排除**: 99,292行(2025) → `(name, cc, c)` ユニークタプルは大幅に少ない。同一タプルは1回だけ判定しキャッシュ。
- **ゲーティング**: 明白な行はルールで即決しAI呼び出しをスキップ可能（`--gate` オプション）。不透明キーワード一致→identifiability=0、辞書valid＋CN＋具体的cc→identifiability=3。曖昧な中間のみAIへ。
- **永続キャッシュ**: `data/cache/ai-quality-cache.json`（コンテンツハッシュ键）。再実行・年度追加は差分のみ課金。`.gitignore` 管理。
- **モデル**: OpenRouter（OpenAI互換, `openai` SDK）。`--model` で選択、デフォルト `google/gemini-3.5-flash`（`anthropic/claude-haiku-4-5`, `openai/gpt-4o-mini` 等も可）。`--base-url` でゲートウェイ変更可。
  - ※ 専用Batch APIは未配線（`--batch` は同期チャンク呼び出しにフォールバック）。コスト制御はキャッシュ＋ゲーティング＋並列（`--concurrency`）＋flash系モデル。
  - ※ キャッシュ名はモデル末尾セグメントで正規化するため、同一実モデルならプロバイダを跨いでも既存キャッシュを再利用（例: 直Gemini→OpenRouter経由）。

### APIキー無し時のフォールバック

`OPENROUTER_API_KEY` 未設定時は**決定的ヒューリスティック判定**で全行を採点し、出力に `aiSource: "heuristic"` を付与（実AI結果と区別）。これによりパイプラインが常に完走し、ローカル検証が可能。本番は実LLMで `aiSource: "openrouter:<model>"`。

## 8. 追補（2026-06-16）: 有効性軸の統合 — 透明性に加え「国民生活への寄与」

透明性（追跡可能性）だけでなく、**事業が国民生活にどれだけ寄与するか**の評価も求められたため、5軸目を追加。

### 方針（ユーザー決定）
- **根拠データ**: 既存 `rs{year}-project-details.json`（目的・現状課題・概要）からAIが判定。生CSV不要で即構築可。
  - ⚠️ これは**実測成果ではなく「成果設計の明確さ（意図ベース）」**。誰のどんな課題を、どの活動で、どう改善するかが明確・妥当に説明されているかを測る。実測成果（アウトカム指標の達成度・点検結果）による評価は、RS 3系/4系CSVの抽出が必要な将来拡張。
- **スコア扱い**: 既存総合に**統合**（独立2軸併記ではなく一本化）。

### 重み（最終）
| 軸 | 種別 | 重み |
|----|------|------|
| A 特定可能性 | AI | 28% |
| B 使途の説明性 | AI | 22% |
| C 収支の整合性 | 機械 | 15% |
| **E 有効性／成果設計の明確さ** | AI | **35%** |
| D 構造の整合性 | 機械 | （参考・不算入） |

有効性をユーザーが重視したため単独最大の35%。A・B・E=AI(85%)・C=機械(15%)。
※ 当初は5軸均等配分（25/20/15/10/30）だったが、軸D（構造）が平均99.9でほぼ弁別しないため**総合から除外し参考表示のみ**に変更、その分をA/B/Eへ再配分。

### 実装
- 軸Eは**事業単位**でのAI判定（per-recipientではない）。`effect_key`（目的+課題+概要のhash）で重複排除しキャッシュ。
- **レベル0-10の11段階**（→ スコア = レベル×10、0-100）。`effectiveLevel`(0-10)・`effectiveReason` を出力に付与。
  - ※ 当初0-3だったが粗すぎ（軸Eヒストグラムが4値に固着）たため0-10へ変更。`effect_key` を `E:`→`E10:` にして再判定（支出先A/B判定は再利用）。
- ヒューリスティックフォールバック: 便益キーワード・数値・本文長から概算（実AI未実行時の土台）。
- 出力フィールド: `axisEffective`・`effectiveLevel`・`effectiveReason`。UIは一覧「有効性」列＋ダイアログにレベル・根拠・「※実測成果ではない」注記を表示。
